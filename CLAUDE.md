# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What is DocketMind

An AI-powered Discord bot that tracks federal lawsuits via CourtListener RSS feeds. When a new docket entry arrives, it downloads any attached PDFs, chunks and embeds everything into ChromaDB, and updates a per-case memory summary. Users ask questions in Discord; the bot answers using RAG + Claude with citations back to the original filings. Admins manage cases via Discord slash commands.

## Commands

```bash
# Install dependencies
uv sync

# Run the bot
uv run python -m docketmind

# Run all tests
uv run pytest

# Run a single test file
uv run pytest tests/ingestion/test_rss.py

# Run a single test by name
uv run pytest -k "test_parse_entry"

# Lint and format
uv run ruff check .
uv run ruff format .

# Type check
uv run pyright

# Database migrations
uv run alembic upgrade head         # apply migrations
uv run alembic revision --autogenerate -m "description"  # generate new migration
```

## Architecture

Three internal packages, strictly layered — bot depends on intelligence and ingestion, they do not depend on bot:

**`docketmind/ingestion/`** — everything that brings data in: CourtListener RSS polling (APScheduler + feedparser + httpx), PDF downloading (httpx + aiofiles), LlamaIndex document loading, chunking, embedding (OpenAI `text-embedding-3-small`), LlamaIndex SimpleVectorStore indexing (disk-persisted), and per-case memory updates. All retries use stamina.

**`docketmind/intelligence/`** — everything that answers questions: LlamaIndex RAG query engine over ChromaDB, per-case memory retrieval, OpenAI LLM calls via LlamaIndex, citation formatting.

**`docketmind/bot/`** — platform adapters only. `base.py` defines the abstract interface all platforms must implement. `discord/` is the only active adapter. Future platforms (Slack, Telegram) add a new subdirectory implementing the same interface without touching ingestion or intelligence.

**`docketmind/config.py`** — single pydantic-settings `Settings` class, loaded from `.env`. All other modules import settings from here, never from `os.environ` directly.

## Data Layout

All persistent state lives in `data/` (gitignored):
- `data/docketmind.db` — SQLite database (cases, docket entries, metadata)
- `data/index/` — LlamaIndex vector store (persisted JSON)
- `data/pdfs/` — downloaded PDF files, organized by `case_id/`

SQLAlchemy models define the schema; Alembic manages migrations in `alembic/`.

## Code Style

- Always add docstrings to all modules, classes, methods, and functions using standard Python triple-quoted strings
- Module docstring at the top of every file (one line summary)
- Class docstring explains purpose and any important behavioral notes
- Method/function docstrings on all public callables; one-liners are fine when the signature is self-explanatory

## Key Conventions

- Everything async — discord.py, httpx, aiofiles, SQLAlchemy async session, APScheduler async jobs
- Pydantic v2 models for all data validation and config; never use plain dicts at module boundaries
- Bot adapters translate platform events into domain calls — no business logic inside `bot/`
- Ingestion is idempotent — re-syncing a case must not create duplicate DB rows or ChromaDB documents (use CourtListener entry ID as dedup key)
- Per-case memory is a single text document regenerated by Claude after each new ingestion batch, stored in SQLite alongside the case record
